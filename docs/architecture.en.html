1 · Architecture

The factory separates code from knowledge: code lives in the system's repositories (cloned under repos/, each with its own git); the shared knowledge — specs, decisions, incidents, rules, maps — lives in this workspace and is versioned here. The system topology is data, not prose: declared once in repos.yaml, everything else is derived from it.

Non-negotiable principles
  • The spec is the contract: every non-trivial change starts as a spec and is certified against it.
  • Human gates backed by permissions, not instructions: agents hold no production credentials.
  • Whatever can be derived, is derived (as-is map, DORA metrics, this document); every published metric declares source and period or says "no data" (rule RN-F4).
  • Strict TDD: failing tests are written and committed first; modifying tests to make them pass is forbidden.
  • Process proportional to risk: the F0 triage decides how much control each change loads (rule RN-F1).

The F0–F9 lifecycle

A requirement flows through ten phases. Each phase is executed by a specialized skill or subagent, and the important decisions stop at a human gate that gets recorded (who, when, on which commit).

The knowledge layers

Four layers answer different questions; together they form the system's memory:

as-is (knowledge/as-is/) — WHAT EXISTS: real state derived from the code, never hand-edited; fine-grained graph in the codebase-memory MCP.
Context packs (loadable skills) — HOW IT IS BUILT: mental model per repo and per system, with file:line evidence, freshness seal and executable assertions.
ADRs (knowledge/decisiones/) — WHY: architecture decisions with context and consequences.
Specs (specs/) — WHAT SHOULD EXIST: each feature's contract, with its gates recorded.

Skills (/… commands)

Derived from .claude/skills/*/SKILL.md; the frontmatter description is the auto-load trigger.

CommandWhat it does (frontmatter)
/as-is-learnAnalyzes the REAL repos to learn how routes are defined and consumed, writes exact per-repo extractors and regenerates the as-is map. Use when the cross-repo graph comes out empty or incomplete, or when a repo's framework changes.
/as-is-syncRegenerates the as-is map from the code and commits the result. Use after structural changes (new modules, endpoints, migrations) or when /as-is detects it is out of date.
/as-isQuery the repository's real as-is map (modules, dependencies, APIs, data). Use when asked how the application is structured, what depends on what, which APIs exist, or before designing (F5).
/clarifyResolves the ambiguities of a requirement or spec with structured questions (max. 5) and records the answers in the spec's Clarifications section. Use between the F0 triage and F1, when asked to "clarify the requirement", or when a spec has [NEEDS CLARIFICATION] markers.
/consistencyRead-only cross-consistency analysis between a spec and the rest of the factory's artifacts (business rules, ADRs, as-is, .claude/rules rules, task plan). Use before the DoR gate and the Architecture gate, or when asked to "analyze consistency" of a spec.
/convergeCloses the functional code↔spec gap at the end of construction (F6/F7) — evaluates the real code against the spec's stories and criteria and ADDS the remaining work as convergence tasks. Use before F7 certification or when asked to "converge the spec" / "what is missing from the spec".
/harness-initPrepares the harness for a large or multi-session feature, derived from its approved spec (Anthropic's initializer pattern).
/implement-taskExecutes ONE task from a spec's plan with strict TDD protocol (F6 Construction). Use to implement tasks one at a time.
/orchestrateWalks the E2E 9-phase cycle for a feature, delegating to the per-phase agents and stopping at each human gate. Only invocable by humans.
/repo-addOnboards a repository into the factory (git URL or local path): clones, registers in repos.yaml, fills its CLAUDE.md with real data, indexes it in codebase-memory and regenerates the as-is map. Use when asked to "add/onboard a repo" or "set up analysis for a repository".
/repo-mapGenerates or updates a repository's CONTEXT PACK (loadable skill .claude/skills/<prefijo>-<repo>) — architecture, core mechanisms, dependencies and pitfalls with file:line evidence — and seeds its assertions. Use when asked to "map a repo", "generate the repo's context", when /spec-create detects a repo without a pack or with a stale pack, or after /repo-add.
/spec-createGenerates a layered feature spec (F1-F5) using the factory template. Use when someone asks to "create spec", "specify feature" or describes new functionality before implementing it.
/spec-reviewValidates a spec against the factory's Definition of Ready. Use before the DoR gate or when asked to "review spec".
/system-mapGenerates or updates the SYSTEM PACK (<prefijo>-sistema, the junctions and the mental model that live in no repo) and the mapa-sistemas index. Use when the system has ≥2 repos with packs, when asked for "the system map"/"how the repos connect", or when /spec-create detects the system pack is missing.

Enforcement and CI

  • Permissions over instructions: settings.json denies agents credentials (.env, keys), deployments (kubectl, terraform apply) and direct pushes to main. Every skill declares least-privilege allowed-tools.
  • CI (bitbucket-pipelines.yml / GitHub Actions template): on every merge to main and on a daily schedule it clones the registry's repos, regenerates as-is + DORA metrics + this document, and commits if anything changed.
  • Context verification: frescura.sh detects stale packs by commit seal; afirmaciones.sh runs each pack's assertions (detects WHICH claim became false).

Artifacts and traceability

Every feature leaves a verifiable end-to-end chain:

Spec specs/*.md→ ADRs knowledge/decisiones/→ Commits (one per task, in its repo)→ CAB dossier→ Postmortems knowledge/incidentes/→ uso.md (DORA) → feeds F1/F2

Specs use the 12-section template (specs/_template.md): problem · objective · measurable success criteria SC-xx · out of scope · traced clarifications · P1–P3 stories with Gherkin · WSJF estimation · analysis (rules, dependencies, edge cases, assumptions) · design (contracts, C4, ADRs, STRIDE, NFRs) · task plan tagged per repo · certification · traceability. Pending [NECESITA CLARIFICACIÓN] markers block the DoR (rule RN-F3).

4 · Usage guide

Scenario A — Install the factory for a new system

git clone https://github.com/mmaripanguec/dev-sdd-kit.git my-system && cd my-system
./scripts/init-system.sh my-system   # cleans instance data, creates the registry
/repo-add <url-or-path>            # once per system repo
/system-map                        # once ≥2 repos have context packs

Scenario B — New machine for an existing system

git clone <workspace-url> && cd <workspace>
cp .env.example .env               # credentials per git provider
./scripts/setup.sh                 # clones/updates ALL registry repos

Scenario C — Specify and build a feature

/spec-create <name> <description>     # F0 triage + F1..F5 with gates
/clarify <spec>                    # if ambiguous (max 5 questions)
/spec-review specs/<spec>.md          # 13-point Definition of Ready
/consistency specs/<spec>.md         # cross-check; needs APTO PARA GATE
/harness-init specs/<spec>.md         # only for multi-session work
/implement-task specs/<spec>.md T1    # TDD, one commit per task IN its repo
/converge specs/<spec>.md            # code↔spec gaps before certifying

Golden rule: always launch Claude Code from the workspace root; code commits go inside each repo (git -C repos/<repo> …), knowledge commits go here.

Scenario D — Query and operate

/as-is which endpoints does X expose?   # real state (graph + map)
/as-is-sync                             # after structural changes
./scripts/dora.sh                       # recompute metrics (F9, after incidents)
/orchestrate specs/<spec>.md              # full E2E cycle (humans only)
