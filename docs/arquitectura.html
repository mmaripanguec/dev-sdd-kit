1 · Arquitectura

La fábrica separa código de conocimiento: el código vive en los repos del sistema (clonados en repos/, cada uno con su propio git); el conocimiento compartido — specs, decisiones, incidentes, reglas, mapas — vive en este workspace y se versiona aquí. La topología del sistema es dato, no prosa: se declara una sola vez en repos.yaml y todo lo demás se deriva.

Principios no negociables
  • La spec es el contrato: todo cambio no trivial parte de una spec y se certifica contra ella.
  • Gates humanos respaldados por permisos, no solo por instrucciones: los agentes no tienen credenciales de producción.
  • Lo derivable se deriva (as-is, métricas DORA, este documento); toda métrica declara fuente y período o declara «sin datos» (RN-F4).
  • TDD obligatorio: el test se escribe y commitea antes; prohibido modificar tests para hacerlos pasar.
  • El proceso es proporcional al riesgo: el triage F0 decide qué controles cargar (RN-F1).

El ciclo de vida F0–F9

Un requerimiento recorre diez fases. Cada fase la ejecuta una skill o un subagente especializado y las decisiones importantes se detienen en un gate humano que queda registrado (quién, cuándo, sobre qué commit).

FaseNombreLa ejecutaProduceGate

Las capas de conocimiento

Cuatro capas responden preguntas distintas; juntas forman la memoria del sistema:

as-is (knowledge/as-is/) — QUÉ HAY: estado real derivado del código, nunca editado a mano; grafo fino en el MCP codebase-memory.
Packs de contexto (skills cargables) — CÓMO ESTÁ ARMADO: modelo mental por repo y de sistema, con evidencia archivo:línea, sello de frescura y aserciones ejecutables.
ADRs (knowledge/decisions/) — POR QUÉ: decisiones de arquitectura con contexto y consecuencias.
Specs (specs/) — QUÉ DEBERÍA HABER: el contrato de cada feature, con sus gates registrados.

Skills (comandos /…)

Derivadas de .claude/skills/*/SKILL.md; la descripción del frontmatter es el gatillo de auto-carga.

ComandoQué hace (frontmatter)
/as-is-learnAnalyzes the REAL repos to learn how routes are defined and consumed, writes exact per-repo extractors and regenerates the as-is map. Use when the cross-repo graph comes out empty or incomplete, or when a repo's framework changes.
/as-is-syncRegenerates the as-is map from the code and commits the result. Use after structural changes (new modules, endpoints, migrations) or when /as-is detects it is out of date.
/as-isQuery the repository's real as-is map (modules, dependencies, APIs, data). Use when asked how the application is structured, what depends on what, which APIs exist, or before designing (F5).
/clarifyResolves the ambiguities of a requirement or spec with structured questions (max. 5) and records the answers in the spec's Clarifications section. Use between the F0 triage and F1, when asked to "clarify the requirement", or when a spec has [NEEDS CLARIFICATION] markers.
/consistencyRead-only cross-consistency analysis between a spec and the rest of the factory's artifacts (business rules, ADRs, as-is, .claude/rules rules, task plan). Use before the DoR gate and the Architecture gate, or when asked to "analyze consistency" of a spec.
/convergeCloses the functional code↔spec gap at the end of construction (F6/F7) — evaluates the real code against the spec's stories and criteria and ADDS the remaining work as convergence tasks. Use before F7 certification or when asked to "converge the spec" / "what is missing from the spec".
/harness-initPrepares the harness for a large or multi-session feature, derived from its approved spec (Anthropic's initializer pattern).
/implement-taskExecutes ONE task from a spec's plan with strict TDD protocol (F6 Construction). Use to implement tasks one at a time.
/orchestrateWalks the E2E 9-phase cycle for a feature, delegating to the per-phase agents and stopping at each human gate. Only invocable by humans.
/repo-addOnboards a repository into the factory (git URL or local path): clones, registers in repos.yaml, fills its CLAUDE.md with real data, indexes it in codebase-memory and regenerates the as-is map. Use when asked to "add/onboard a repo" or "set up analysis for a repository".
/repo-mapGenerates or updates a repository's CONTEXT PACK (loadable skill .claude/skills/<prefijo>-<repo>) — architecture, core mechanisms, dependencies and pitfalls with file:line evidence — and seeds its assertions. Use when asked to "map a repo", "generate the repo's context", when /spec-create detects a repo without a pack or with a stale pack, or after /repo-add.
/spec-createGenerates a layered feature spec (F1-F5) using the factory template. Use when someone asks to "create spec", "specify feature" or describes new functionality before implementing it.
/spec-reviewValidates a spec against the factory's Definition of Ready. Use before the DoR gate or when asked to "review spec".
/system-mapGenerates or updates the SYSTEM PACK (<prefijo>-sistema, the junctions and the mental model that live in no repo) and the mapa-sistemas index. Use when the system has ≥2 repos with packs, when asked for "the system map"/"how the repos connect", or when /spec-create detects the system pack is missing.

Enforcement y CI

  • Permisos sobre instrucciones: settings.json niega a los agentes credenciales (.env, llaves), despliegues (kubectl, terraform apply) y push directo a main. Cada skill declara allowed-tools de mínimo privilegio.
  • CI (bitbucket-pipelines.yml / plantilla GitHub Actions): en cada merge a main y en un schedule diario clona los repos del registro, regenera as-is + métricas DORA + este documento y commitea si hay cambios.
  • Verificación del contexto: freshness.sh detecta packs viejos por sello de commit; assertions.sh ejecuta las aserciones de cada pack (detecta QUÉ afirmación se volvió falsa).

3 · Modelo

Diagrama de flujo conceptual

La spec es el contrato que gobierna el flujo; el conocimiento que la operación produce alimenta la siguiente feature — el ciclo se cierra:

Artefactos y trazabilidad

Cada feature deja una cadena verificable de punta a punta:

Spec specs/*.md→ ADRs knowledge/decisions/→ Commits (uno por tarea, en su repo)→ Expediente CAB→ Postmortems knowledge/incidents/→ usage.md (DORA) → alimenta F1/F2

La spec usa la plantilla de 12 secciones (specs/_template.md): problema · objetivo · criterios de éxito SC-xx medibles · fuera de alcance · clarificaciones trazadas · historias P1–P3 con Gherkin · estimación WSJF · análisis (RN, dependencias, casos límite, supuestos) · diseño (contratos, C4, ADRs, STRIDE, NFRs) · plan de tareas etiquetadas por repo · certificación · trazabilidad. Los marcadores [NECESITA CLARIFICACIÓN] pendientes bloquean la DoR (RN-F3).

4 · Guía de uso

Escenario A — Instalar la fábrica para un sistema nuevo

git clone https://github.com/mmaripanguec/dev-sdd-kit.git mi-sistema && cd mi-sistema
./scripts/init-system.sh mi-sistema   # limpia datos de instancia, crea registro
/repo-add <url-o-ruta>             # por cada repo del sistema
/system-map                        # cuando haya ≥2 repos con pack

Escenario B — Máquina nueva de un sistema existente

git clone <url-del-workspace> && cd <workspace>
cp .env.example .env               # credenciales según proveedor
./scripts/setup.sh                 # clona/actualiza TODOS los repos del registro

Escenario C — Especificar y construir una feature

/spec-create <nombre> <descripción>   # triage F0 + F1..F5 con gates
/clarify <spec>                    # si hay ambigüedad (máx. 5 preguntas)
/spec-review specs/<spec>.md          # DoR de 13 puntos
/consistency specs/<spec>.md         # cruzada; APTO PARA GATE para avanzar
/harness-init specs/<spec>.md         # solo si es multi-sesión
/implement-task specs/<spec>.md T1    # TDD, un commit por tarea EN su repo
/converge specs/<spec>.md            # brechas código↔spec antes de certificar

Regla de oro: lanzar Claude Code SIEMPRE desde la raíz del workspace; los commits de código van dentro de cada repo (git -C repos/<repo> …), los de conocimiento aquí.

Escenario D — Consultar y operar

/as-is ¿qué endpoints expone X?    # estado real (grafo + mapa)
/as-is-sync                        # tras cambios estructurales
./scripts/dora.sh                  # recalcular métricas (F9 tras incidente)
/orchestrate specs/<spec>.md         # ciclo E2E completo (solo humano)
